ДОКУМЕНТАЦИЯ ПО ПРОЕКТУ: DATA INGESTION PIPELINE ДЛЯ RAG-СИСТЕМЫ
====================================================================

ОПИСАНИЕ ПРОЕКТА
----------------
Данный проект представляет собой высокопроизводительный пайплайн подготовки данных (Data Ingestion) для базы знаний по спортивной фармакологии. Цель пайплайна — принимать разнородные источники данных (PDF, DOCX, XML, веб-страницы), извлекать из них чистый, структурированный текст и приводить к единому формату для последующей загрузки в векторную базу данных (FAISS) и использования ИИ-агентом (RAG).

ЖЕСТКИЕ ОГРАНИЧЕНИЯ (Соблюдены на 100%)
-------------------------------------
1. Выполнение строго на CPU в условиях ограниченной оперативной памяти.
2. Отказ от тяжелых ML-моделей и OCR (Nougat, Marker и т.д.). Использование исключительно легковесных эвристических алгоритмов парсинга.

ТЕКУЩИЙ СТАТУС
--------------
Этапы 1 (Router), 2 (Extractors) и 3 (Normalization) полностью завершены. Код написан на уровне Senior Python Developer: строгая типизация (Type Hints), защита от сбоев (try/except), модульная структура и подробное логирование.


АРХИТЕКТУРА И РЕАЛИЗАЦИЯ
====================================================================

ЭТАП 1: МАРШРУТИЗАТОР (Document Router)
Файл: router.py
--------------------------------------------------------------------
Является единой точкой входа (API). Принимает строку (локальный путь или URL) и интеллектуально направляет поток в нужный парсер.

Ключевые механизмы:
1. Защита от OOM (Out-Of-Memory):
   - Внедрена жесткая константа MAX_FILE_SIZE = 50 MB.
   - Для локальных файлов размер проверяется через os.path.getsize().
   - Для URL выполняется предварительный запрос заголовков (requests.get с stream=True), размер считывается из Content-Length до начала загрузки.

2. Сниффинг MIME-типов и Магические байты:
   - Маршрутизатор не слепо доверяет расширениям файлов. Если у файла нет расширения или оно неизвестно, запускается проверка "магических байт" (первых 4 байт файла).
   - Например, сигнатура `%PDF` гарантированно отправляет файл в PDF-парсер, а `PK\x03\x04` (ZIP) — в DOCX.
   - В качестве запасного варианта используется модуль `mimetypes`.

3. Умная обработка URL:
   - Маршрутизатор читает Content-Type сервера. Если ссылка ведет на бинарный файл (например, application/pdf), он безопасно выкачивается кусками во временный файл (через модуль tempfile).
   - Временный файл скармливается локальному парсеру, после чего гарантированно удаляется. Метаданные (source) при этом подменяются на оригинальный URL.
   - Обычные веб-страницы (text/html) отправляются в HTML-парсер.


ЭТАП 2: ИЗОЛИРОВАННЫЕ ЭКСТРАКТОРЫ (Parsers)
Файл: extractors.py
--------------------------------------------------------------------
Модуль содержит 4 изолированные функции. Каждая имеет защиту от сбоев и возвращает унифицированный словарь (Этап 3).

1. PDF Парсер (parse_pdf)
   Библиотека: PyMuPDF (fitz)
   - Умная сортировка колонок: Используется встроенный алгоритм `page.get_text("blocks", sort=True)`, который восстанавливает естественный порядок чтения даже при сложной двухколоночной верстке.
   - Склейка разорванных слов (Дефисация): Регулярное выражение находит слова, разорванные дефисом на новую строку (например, "фарма-\nкология"), и склеивает их в целые слова для точного RAG-поиска.
   - Конвертация таблиц: Найденные на странице таблицы автоматически преобразуются в Markdown-формат (через .to_markdown() или вручную), что критически важно для сохранения дозировок препаратов. Блоки, пересекающиеся с таблицей, исключаются из обычного потока во избежание дублирования.
   - Отсечение мусора: Парсинг жестко останавливается, как только регулярное выражение находит раздел "References" / "Bibliography" (в т.ч. с нумерацией). Колонтитулы отсекаются по Y-координатам (верхние/нижние 5%).

2. DOCX Парсер (parse_docx)
   Библиотека: python-docx
   - Сохранение порядка элементов: Для предотвращения смешивания контекста парсер выполняет низкоуровневый обход XML-дерева (`doc.element.body`).
   - Текст параграфов и Markdown-таблицы извлекаются строго в той последовательности, в которой они расположены в документе.

3. XML Парсер (parse_xml)
   Библиотека: BeautifulSoup4 (bs4, парсер lxml)
   - Адаптация под научные JATS XML. Мусорные теги (<ref-list>, <back>) вырезаются полностью.
   - Внедрение Markdown: Извлекаемые заголовки (<article-title>, <abstract>, внутренние <title> секций) оборачиваются в Markdown (`# `, `## `, `### `). Это создает иерархию, которая позволит LangChain MarkdownTextSplitter на Этапе 4 умно нарезать документ без разрыва мыслей.

4. HTML Парсер (parse_html)
   Библиотеки: requests, readability-lxml, bs4
   - Для скачивания используется поддельный User-Agent, чтобы научные журналы не отдавали ошибку 403 Forbidden.
   - Библиотека Mozilla Readability математически вычисляет главный текстовый блок (article), полностью отрезая рекламные баннеры, боковые панели, меню навигации и футеры.
   - BeautifulSoup выполняет финальную очистку, оставляя только сырой, полезный текст статьи.


ЭТАП 3: НОРМАЛИЗАЦИЯ (Normalization)
--------------------------------------------------------------------
Реализовано на уровне выходов всех 4-х парсеров.
Все функции строго возвращают словарь (или None в случае ошибки/сбоя):

{
    "text": "Отфильтрованный, чистый текст (включая склеенные слова и таблицы в Markdown)",
    "metadata": {
        "source": "Оригинальное имя файла или URL-адрес",
        "format": "Тип файла (pdf, docx, xml, html, url)"
    }
}
