# OmniRAG: Точный локальный AI-ассистент

Проект представляет собой оптимизированную локальную RAG-систему (Retrieval-Augmented Generation), которая позволяет вести диалог по содержимому пользовательских документов (PDF, DOCX, TXT, HTML, XML). Система разработана с прицелом на строгую фактологию и работает максимально стабильно даже на небольших языковых моделях (например, Qwen 2.5 3B).

### Основные архитектурные особенности
- **Режим "Строгого редактора"**: LLM-модель работает с температурой 0.0 (нулевая креативность), что исключает галлюцинации и выдумывание фактов. Ответы генерируются строго по найденному тексту.
- **Продвинутый поиск (Hybrid Search + Reranker)**: В качестве векторной базы используется **Qdrant**. Система делает гибридный поиск (по смыслу и по ключевым словам BM25), извлекая 15 релевантных кусков. Затем сверхлегкий **Cross-Encoder** (работающий даже на CPU) перепроверяет их и оставляет только 4 самых точных куска. Это защищает модель от проблемы "Lost in the Middle" и не допускает потерю данных.
- **Обогащение метаданными (Metadata Enrichment)**: Имя исходного документа "вшивается" в каждый его текстовый фрагмент, что в разы повышает точность поиска по узким запросам (например, когда вопрос касается конкретной книги или автора).
- **Двухуровневая память**: Qwen 2.5 3B сжимает диалог в summary из 3–5 предложений. Активное summary хранится в Redis с TTL, а долговечная копия каждой сессии — в PostgreSQL.
- **Обход графа в Qdrant**: найденные векторным поиском чанки становятся начальными узлами ограниченного BFS. Соседи из `graph_edges`, `related_ids` и связей соседних чанков добавляются к кандидатам, после чего проходят через существующий reranker.
- **Временные файлы**: Возможность загружать документы через пользовательский веб-интерфейс только на время текущей сессии. Как только страница перезагружается, эти данные удаляются из оперативной памяти.
- **Полностью локальная обработка**: Вся обработка (эмбеддинги, векторизация, генерация Ollama) происходит на устройстве пользователя без передачи данных в интернет. Код автоматически задействует GPU (видеокарту) при наличии, а при отсутствии плавно переключается на CPU.

## Краткое руководство по запуску

Проект разворачивается с помощью Docker и использует профили для определения вычислительных ресурсов (CPU или GPU). 

### 1. Подготовка документов
Поместите ваши файлы с текстом, которые должны войти в глобальную базу знаний, в папку `data/` в корневой директории проекта.

### 2. Запуск контейнеров
Откройте терминал в папке с проектом и выполните команду запуска в зависимости от конфигурации оборудования.

Для запуска на процессоре (CPU):
```bash
docker-compose --profile cpu up -d
```

Для запуска с использованием видеокарты (GPU):
```bash
docker-compose --profile gpu up -d
```

### 3. Загрузка языковой модели
Контейнер с нейросетью скачивается без встроенной модели. При первом запуске необходимо загрузить нужную модель вручную (рекомендуется модель на 3B параметров):
```bash
docker exec ollama_db ollama pull qwen2.5:3b
```

### 4. Индексация базы данных
Для того чтобы система распознала файлы, помещенные в папку `data/`, необходимо запустить скрипт векторизации. 

Если вы запускали профиль CPU:
```bash
docker exec -it pharma_rag_ui_cpu python -m src.cli.ingest
```

Если вы запускали профиль GPU:
```bash
docker exec -it pharma_rag_ui_gpu python -m src.cli.ingest
```

При добавлении новых файлов в папку `data/` эту команду необходимо повторять. Система автоматически определит и добавит только новые документы. После обновления до схемы индекса v2 первый запуск один раз переиндексирует старые документы, чтобы добавить `chunk_id` и графовые связи.

### Память и graph traversal

При запуске профиля Docker автоматически поднимаются два дополнительных сервиса:

- `redis` — горячее summary активной вкладки; стандартный TTL — 1800 секунд;
- `postgres` — холодный архив summary по `user_id` и `session_id`.

В боковой панели задается идентификатор пользователя. При новом входе последнее summary читается из PostgreSQL и прогревается в Redis. Кнопка «Завершить сеанс» сразу удаляет горячую копию, не затрагивая архив.

Qdrant хранит в payload узлы и ребра, а BFS выполняет приложение. По умолчанию используются коллекция `omni_rag_collection`, глубина 1 и не более 6 дополнительных узлов. Настройки можно изменить через `.env` по примеру `.env.example`:

```dotenv
GRAPH_TRAVERSAL_DEPTH=1
GRAPH_TRAVERSAL_MAX_NODES=6
SESSION_MEMORY_TTL_SECONDS=1800
```

Собственные узлы графа могут использовать поля `metadata.node_id` или `metadata.chunk_id`, а ребра — формат `graph_edges: [{"target_id": "...", "relation": "..."}]`. Также поддерживаются `related_ids`, `prev_chunk_id` и `next_chunk_id`.

### 5. Использование интерфейса
После завершения процесса индексации приложение будет доступно в браузере по адресу:
http://localhost:8501

*(Также в чате можно прикрепить любой временный файл, чтобы нейросеть прочитала его прямо на лету).*

### Остановка проекта
Для завершения работы всех контейнеров выполните команду:
```bash
docker-compose --profile cpu down
```
*(Замените `cpu` на `gpu`, если использовали этот профиль).*
